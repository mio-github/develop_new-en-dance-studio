## 🔧 システム設計に反映すべきポイント一覧

---

### ① 会員情報・認証設計

| 項目             | 内容                           | 必須度 |
| -------------- | ---------------------------- | --- |
| メールアドレス＝ログインID | 再登録時の識別キー、重複・変更対策も必要         | ★★★ |
| 会員番号（旧ID）との紐付け | 過去データとのリレーションのために保持          | ★★★ |
| メンバーID（新ID）管理  | 箱物で自動採番されるので注意               | ★★★ |
| 同一人物判定機能       | メールアドレス変更で再登録された場合などの照合      | ★★☆ |
| 強制退会・フラグ制御     | ブラックリストの代替機能として運用（例：受講不可フラグ） | ★★☆ |

---

### ② 決済周りの仕様設計

| 項目              | 内容                              | 必須度 |
| --------------- | ------------------------------- | --- |
| クレジットカード決済（GMO） | 月謝支払いの主な手段。未納時は口座振替へ            | ★★★ |
| 口座振替（ジャックス）     | クレカNG時のバックアップ決済。3ヶ月未納で強制退会処理    | ★★★ |
| 決済状態の自動遷移       | クレカ失敗→口座振替→店頭支払（コンビニ/LINE Pay等） | ★★★ |
| ジャックス/GMO 並行運用  | 顧客に再登録の手間なし。メンテも考慮した連携設計        | ★★☆ |
| Airペイ利用（店頭決済）   | 現金ではなく、電子マネー中心                  | ★☆☆ |
| クレピコ（物理端末）依存回避  | 店頭不要で完結するような構成推奨                | ★★☆ |

---

### ③ チケット・QR関連

| 項目         | 内容                           | 必須度 |
| ---------- | ---------------------------- | --- |
| QRコードによる受付 | 毎回発行（使い回し不可）、読み取り必須          | ★★★ |
| 電子チケット発行   | 公演やレッスン時に発行し、利用者が表示          | ★★★ |
| 同時アクセス集中対応 | 発表会前後などに集中アクセス想定（最大1000人/1h） | ★★★ |
| 待機・順番処理設計  | サーバー負荷回避のため、分散/非同期設計も検討      | ★★☆ |

---

### ④ データ移行と互換性対応

| 項目           | 内容                  | 必須度 |
| ------------ | ------------------- | --- |
| CSVベースの手動移行  | 現システムからのデータ抽出はCSVが主 | ★★★ |
| 会員情報7〜8割は取得可 | 残りは漏れ項目を再確認する必要あり   | ★★☆ |
| 旧会員番号・旧履歴保持  | 分析・検索・問い合わせのために残す設計 | ★★★ |
| API連携可能性の調査  | 現在は別料金。将来必要なら内製検討   | ★☆☆ |

| -------------- | ------------------------ | --- |

| スタッフ主導でのフォロー機能 | 電話やLINEで未ログイン者へ個別連絡      | ★★☆ |
